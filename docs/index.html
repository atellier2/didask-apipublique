API Publique Didask

Introduction

L’API Didask vous permet principalement de gérer l'organisation des utilisateurs de votre application Didask et de consulter l'avancement de l'apprentissage des apprenants.

Si vous souhaitez gérer les publications (=contenus pédagogiques), il est nécessaire de passer par le système SCORM également géré par Didask. L'API publique ne sera pas la bonne solution.

Exemple courant d'usage de l'API Publique :

  • Création des utilisateurs
  • récupération de l'avancement des l'apprentissage

Documentation fonctionnelle

Espaces de travail

L’application Didask est organisée en espaces de travail. Ces espaces permettent d'organiser les groupes d'utilisaturs, les publications, etc.

💡 L'espace de travail est un élément structurant de la plateforme Didask. Rapprochez vous de votre consultant Didask pour les utiliser à bon essient.

L'API vous permet de réaliser les principales opérations sur les espaces de travail (Consultation, création, modification et suppression).

Publications

Les publications sont les contenus pédagogiques organisés qui sont mis à disposition des apprenants.

L'API vous permet de les consulter mais pas de les modifier. Pour interfacer les publications, préférez l'utilisation du SCORM Connect

Groupe d'utilisateur

Les groupes d'utilisateur permettent de donner l'accès à certaines publications à des utilisateur en définissant le rôle qu'ils jouent vis-à-vis de ces publications..

Les rôles possibles sont : 

- learner : apprenant
- coach : animateur
- author : concepteur
- admin : administrateur

Les groupes sont définis à l’intérieur d’un espace de travail et ne peuvent appartenir qu’à un seul espace de travail.

Deux groupes de travail ne peuvent pas avoir le même nom et ce, même s’ils sont dans deux espaces de travail différents.

Un groupe utliisateur peut avoir une date d'expiration qui est une date optionnelle à partir de laquelle les apprenants n'auront plus accès aux publications.

Utilisateurs

Un utilisateur peut appartenir à plusieurs espaces de travail mais il ne peut avoir le rôle d'apprenant que dans un seul espace de travail.

La clé d’unicité de l’utilisateur est donnée par son adresse mail. Il n’est pas possible de créer deux utilisateurs avec deux id différents et le même email.

Documentation technique

Environnement

Dès lors que vous avez un environnement Didask vous avez accès à l'API publique.

Cette API est disponible à l'adresse votrenom.didaks.com/api/v1

Authentification

L'authentification se fait à l'aide de l'API votrenom.didaks.com/api/login/auth.

💡Nottez qu'il n'y a pas le v1 dans l'URL

L'authentification peut être réalisée par login/mot de passe ou via l'authentification SSO.

Seuls les utilisateurs ayant le rôle d'enseignant peuvent utiliser cette API.

Dans les deux cas, l'API vous renverra un cookies contenant la clé sid. Le cookie sid doit ensuite être inclus dans les requêtes pour accéder aux API.

**Request Headers**
User-Agent: xxxxxxx
Cookie: sid=670e24927e3b774eff8bf73e
...

Cette authentification est valable 1 journée (24h).

Identifiants uniques

Chaque ressource (publication, utilisateur, groupe, espace, etc.) est identifiée de manière unique par un id qui est généré uniquement par Didask et qui est non modifiable. Cet ID est une chaîne de 12 octets (ou 24 caractères hexadécimaux).

Exemple : ertab7234eb4vbnd2e943501

Retour des API

La plus part des API retournent les informations sous le format suivant : 

{
    // Données que vous avez demandées
    "data": [
        //Les données sont organisées en une liste  avec une
        //ou plusieurs valeurs en fonction du type de demande.
        {
            "cle": "valeur",
            "cle2": "valeur2",
            ...
        }
    ],

    //Dans le cas où il y a une erreur, le bloc suivant sera retourné sinon 
    //la clé error prendra la valeur **null**
    "error": {
        "id": "f1c8711f-0a0f-4869-8922-3f41f2e26d82",
        "message": "Argument passed in must be a string of 12 bytes or a string of 24 hex characters or an integer",
        "code": "error.unknown",
        "status": 500,
        "cause": {
            "message": "Complément d'informations"
        }
    }
}

Codes retour

De manière générale, les codes retours utilisés sont les codes standards :

  • 2XX : l'appel a réussi
  • 500 : l'appel a échoué. C'est principalement le cas quand les données communiquées n'ont pas le bon format

Tests de l'API

Pour tester l'API, vous pouvez utiliser un outil tel que Postman ou curl. La spécification est librement téléchargeable (en haut de la page). Cette spécification peut être importée dans Postman (https://learning.postman.com/docs/getting-started/importing-and-exporting/importing-from-swagger/)

Authentification

Comment se connecter à Didask ?

Authentification

  • Appelez l'endpoint POST d'authentification de cette API avec votre email et mot de passe de la plateforme Didask.

  • Vous obtiendrez la valeur d'un cookie appelé sid (identifiant de session) dans les en-têtes de la réponse. Ensuite,

  • Ajoutez le sid récupéré en tant que 'Cookie sid=' dans l'en-tête de toute requête que vous souhaitez envoyer à cette API publique.

💡 Notez que seuls les utilisateurs ayant le rôle d'enseignant peuvent utiliser cette API.

🚨 : cette API est la seule qui ne prend pas comme racine l'URL /api/v1 mais directement auth/login.

Utilisateur

Gestion des utilisateurs de la plateforme Didask

Liste des utilisateurs et leurs accès aux publications.

Cette API permet de lister les utilisateurs Didask en filtrant par nom ou email.

Par défaut, seuls 10 utilisateurs sont retournés. Vous pouvez ajuster l'index de pagination pour accéder aux utilisateurs suivants. Le nombre d'utilisateurs par page est paramétrable.

Pour chaque utilisateur, les informations détaillées ainsi que la liste des publications auxquelles il a accès sont renvoyées.

Note : Cette API est limitée à un seul espace de travail par appel.

Authorizations:
CookieAuth
query Parameters
limit
integer
Default: 10
Example: limit=10

Nombre d'utilisateurs à retourner (0 = illimité). Par défaut, 10 utilisateurs sont retournés.

skip
integer
Default: 0

Index du premier utilisateur à retourner pour la pagination (décalage).

spaceId
string
Example: spaceId=65305dfc128086ef68c5d03f

Filtrer par ID d'espace de travail. Si non renseigné, l'espace de travail par défaut est utilisé.

email
string
Example: email=dupont

Filtrer par une partie de l'email ou du nom d'utilisateur.

string or integer (DidaskId)
Example: id=65305dfc128086ef68c5d03f

Filtrer sur l'ID de l'utilisateur

Création d'un nouvel utilisateur

L'utilisateur sera ajouté à l'espace de travail par défaut si aucun spacesIds n'est spécifié.

Lors de sa création, l'utilisateur a automatiquement un rôle "apprenant" (Learner) qui lui est donné sur l'espace de travail de rattachement.

Soit on utilise l'identification par email/mot de passe, soit on utilise la connexion SSO via oidcIssuer et oidcSubject.

Authorizations:
CookieAuth
Request Body schema: application/x-www-form-urlencoded
required
email
string

Email de l'utilisateur à créer

password
string^(?=.*[a-zA-Z])(?=.*\d)[a-zA-Z-9]{8,}$

Mot de passe en clair de l'utilisateur à créer. Il doit contenir 8 cractères minimum: avec au moins une lettre et un chiffre

firstName
string

Prénom de l'utilisateur à créer

lastName
string

Nom de famille de l'utilisateur à créer

Array of strings or integers (DidaskId)

Liste d'ID des espaces de travail où l'utilisateur sera ajouté. Si aucun espace de travail n'est fourni, l'utilisateur sera créé dans l'espace de travail par défaut.

oidcIssuer
string

Identifiant du fournisseur d'identité OpenID Connect (OIDC) qui a émis le jeton pour cet utilisateur.

oidcSubject
string

Identifiant unique de l'utilisateur dans le cadre de l'authentification OpenID Connect (OIDC).

accessCode
string

Matricule ou code d'accès à la plateforme. (Mode d'authentification alternatif à l'email/password)

Mettre à jour la disponibilité d'un utilisateur.

Cette API peut être utilisée pour activer ou désactiver un Utilisateur

Lorsqu'un utilisateur est désactivé, la date disabledDate est mise à jour automatiquement. De même, lorsqu'un utilisateur est réactivé, la date ``disabledDate` est réinitialisée.

🚨 Lorsqu'un utilisateur est désactivé, il est supprimé de tous les groupes auxquels il appartient et lors de sa réactivation il reste sans groupe.

Remarque : si vous atteignez le nombre maximal de licence pour votre organisation, vous aurez une erreur 403.
